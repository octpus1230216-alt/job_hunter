# 产品说明 · job_hunter 复盘分析模块（Analytics / 反馈闭环）

> 文档类型：产品说明（Product Brief）
> 版本：v1.0 · 日期：2026-07-15 · 状态：已实现并合并进主仓 `analytics/`
> 范围说明：本文档只做产品说明与需求界定，**不含市场分析、竞品调研与可行性验证**（按需求方要求）。

## 1. 一句话定位

**一个"求职错题本 + 简历 A/B 测试台"**：既能在**看到新岗位时**帮你分析匹配度、对照原简历改出针对性简历；也能对**已经投过的岗位**做复盘，反推简历与投递策略的问题，量化"改简历到底有没有用"。

它服务**两个目标**，一条共享底座：

- **主线 A · 前瞻（进攻）**：新岗位进来 → 分析匹配度 → 指出差距 → 改简历 → 导出。
- **主线 B · 复盘（防守）**：已投递结果回流 → 评分 → 交叉分析 + A/B 回复率 → 沉淀改进结论。

它不是"向前海投"工具，而是 job_hunter 主仓"发现职位 → 匹配 → 生成简历 → 投递"链路的**后半段闭环**：从结果回流，让下一轮投递更准；同时把"投递前定点打磨简历"也纳入同一套数据与模型。

## 2. 解决什么问题

产品要同时兜住用户求职链路里的两类动作：

**目标 A（前瞻·新岗位来了怎么办）**
- 看到新 JD，想知道"我到底匹不匹配、差在哪"，而不是凭感觉海投。
- 想照着这个 JD 把原简历改成"更对得上"的版本再投。

**目标 B（复盘·投过的怎么总结）**
用户在 BOSS / 脉脉 大量投递后，普遍出现两类负反馈：

- **主动拒绝（不合适 / rejected）**——信息量大，能定位问题；
- **沉默（已读不回 / no_response）**——弱信号，不能直接归因。

原有「投递追踪」只做状态标记与看板，**不产生可行动洞察，也无法对照岗位改进简历**。本模块让数据"说话"：我的问题集中在哪类岗位？被拒的 JD 跟我简历差在哪？换简历版本后回复率变没变？

两个目标共用同一套底座（数据结构、七维评分、双通道策略、原简历基准），区别只在**入口与终点**：

- 目标 A 的入口是"一条新 JD"，终点是"一份改好的、对照该 JD 的简历"；
- 目标 B 的入口是"一批已投递结果"，终点是"一份复盘结论 + 下一版简历该怎么改"。
- 两者衔接：目标 B 的复盘结论（哪类岗位总被拒、哪版简历回复率高）反哺目标 A，指导下一份简历的打磨方向。

## 3. 产品闭环（两大目标 × 共享底座）

```
┌───────────────────────────── 共享底座（一次准备，两条主线复用）────────────────────────────┐
│  ingest_resume → 原简历基准      store(SQLite)      七维评分模型      双通道策略(免key可跑)  │
└───────────────────────────────────────────────────────────────────────────────────────────┘

主线 A · 前瞻（新岗位 → 分析 → 改简历）【进攻】
   录入新 JD(prospect) ──→ score(对照原简历) ──→ 差距/不足 ──→ revise ──→ export(LaTeX/ATS)
   collect                 score --profile          analyzer/revise     revise      export

主线 B · 复盘（已投递 → 回溯 → 沉淀）【防守】
   收集投递结果 ──→ score ──→ analyzer(统计 / fit×outcome 交叉 / A-B 回复率) ──→ 沉淀改进结论
   collect         score     analyzer                                          (指导下一版简历)

        └── 衔接：B 的复盘结论（哪类岗位总被拒、哪版简历回复率高）反哺 A 的下一版简历打磨 ──┘
```

**前置：原简历上传（ingest_resume，FR9）**——把你的 Word/PDF 原简历解析为文本，作为评分（`--profile`）、差距分析、改简历（FR7）的**基准**。没有原简历，后续"指出不足 / 对照 JD 改简历"无从比对。这是目标 A 的起点，也是目标 B 的参照系。

**主线 A（前瞻）步骤**
1. **录入新 JD（collect）**：把看到的新岗位作为一条 `prospect`（待分析/观望）记录录入，带 `jd_text` / `source_url`。
2. **分析匹配度（score）**：七维评分（技能/经验/文化/职业/背景契合/薪资匹配/层级匹配）加权出总分，对照你上传的原简历。
3. **指出差距（analyzer / revise）**：拿到"JD vs 原简历"的具体差距与不足清单。
4. **改简历（revise）**：对照 JD 生成"修改后简历草稿"（Drafter-Reviewer 思路）。
5. **导出（export）**：简历 LaTeX 生成 + ATS 关键词覆盖率校验，产出可投递版本。

**主线 B（复盘）步骤**
1. **收集投递结果（collect）**：多源录入"我的投递结果"——手动 / CSV / 截图 OCR / 爬虫占位接口，状态含 rejected / no_response / interview / offer 等。
2. **评分（score）**：对每条已投记录打七维分，写入记录。
3. **分析（analyzer）**：基础统计 + fit×outcome 交叉 + JD 差距分析 + 简历版本 A/B 回复率。
4. **沉淀**：从交叉分析与 A/B 里得出"我的问题集中在哪 / 哪版简历更有效"，指导下一轮（目标 A）的简历打磨。

> 两条主线共用 `store` 单一数据源与七维模型；区别仅在入口（新 JD vs 已投结果）与终点（改好的简历 vs 复盘结论）。

## 4. 核心价值主张

| 价值 | 服务于 | 说明 |
|---|---|---|
| 新岗位不盲投 | 目标 A | 看到 JD 先分析匹配度与差距，再决定怎么改简历去投，而不是凭感觉 |
| 把"已读不回"从焦虑变成数据 | 目标 B | 区分强信号（不合适）与弱信号（已读不回），防止"相关性误判为因果"乱改简历 |
| 高亮最该改的信号 | 目标 B | "高 fit + 被拒"= 人岗匹配但简历没讲清 → 改简历；"低 fit + 被拒"= 意料之中，别纠结 |
| 越投越准 | 目标 B→A | 简历版本 A/B 回复率对比，量化"改法有效 / 无效"，反哺下一版打磨 |
| 零门槛可跑 | 共享 | 双通道设计：免 API key 即可跑通全文链路（导出提示词贴 WorkBuddy） |

## 5. 模块全景（含服务目标）

| 引擎 | 文件 | 职责 | 服务目标 |
|---|---|---|---|
| 存储层 | `modules/store.py` | SQLite 数据模型与读写统计（单一数据源） | 共享底座 |
| 原简历上传 | `ingest_resume.py` | 上传并解析原简历（doc/pdf/txt → 文本） | 目标 A（基准）· 目标 B（参照） |
| 收集 | `collect.py` | 多源录入（`add`/`csv`/`image`/`scraper`；新 JD 走 `prospect` 状态） | 两条主线入口 |
| 评分 | `score.py` | 七维匹配度评分（+公司竞争力/真实概率，对照原简历 `--profile`） | 共享 |
| 分析 | `analyzer.py` | 统计 + fit×outcome 交叉 + 差距 + A/B | 目标 B 为主，差距兼顾 A |
| 修改 | `revise.py` | 对照 JD 生成修改草稿 | 目标 A |
| 导出 | `export.py` | LaTeX 生成 + ATS 校验 | 目标 A |

## 6. 适用人群与场景

- **核心用户**：在 BOSS / 脉脉 / 猎聘 等平台主动投递、希望"用数据改进求职"的求职者。
- **典型场景**：投递 20+ 条后复盘、换简历版本前后对比、针对某类"总被拒"岗位定向改简历。
- **不适用**：纯海投且不关心复盘、或要求自动代投的用户（本产品定位"半自动 + 人工决策"）。

## 7. 与 job_hunter 主仓的关系

- 本模块是主仓的**子目录 `analytics/`**，自包含、可独立运行（SQLite + stdlib，零重依赖）。
- 复用主仓理念（多源采集、七维评分、LLM 双通道、LaTeX/ATS 思路），但**聚焦"向后复盘"**，不改动主仓"向前流水线"。
- 合并边界：主仓用 JSON 存储（`data/applications/applications.json`），本模块用 SQLite；长期合并需对齐存储与 `llm.py` 接口（见 DESIGN_SPEC §10）。

## 8. 设计原则

1. **诚实不虚构**：差距与修改建议只基于真实 JD 与原简历，缺的能力明说，绝不编造经历凑关键词。
1. **原简历为基准**：所有"指出不足 / 对照 JD 改简历"都从用户上传的原简历出发，FR9 提供统一上传与 doc/pdf 解析入口。
2. **弱信号不误判**：`no_response`（已读不回）不当作"简历差"的证据，避免相关性误判为因果。
3. **本地优先 / 隐私优先**：所有数据存本地 SQLite，`data/` 经 `.gitignore` 排除，不入库、不上云。
4. **零门槛可跑**：默认模式免 API key；有 key 时一键全自动（双通道）。

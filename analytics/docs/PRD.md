## 2. 背景与问题

用户在 BOSS / 脉脉 大量投递，出现两类负反馈：
- **主动拒绝（不合适 / rejected）**
- **沉默（已读不回 / no_response）**

原「投递追踪」仅做状态标记与看板，**不产生可行动洞察，也无法对照岗位改进简历**。用户原始诉求："把这些数据收集起来分析，最后对照岗位修改。"

此外，"对照岗位改简历"与"指出简历不足"**必须以用户原始简历为基准**——没有原简历，revise/差距分析无从比对。当前录入只覆盖 JD（`jd_text` / `source_url`），**原简历尚缺统一上传入口**（仅能手动粘贴文本）。本文档新增 FR9 补足该缺口：提供上传端口，解析 `.doc/.docx/.pdf/.txt` 等原简历为文本，作为评分、差距分析、改简历的基准。

## 3. 产品目标

本产品同时服务**两个目标**，二者共用同一套数据与模型：

- **目标 A · 前瞻（进攻）**：在看到新岗位时，能分析该 JD 与我的匹配度、指出简历差距，并对照 JD 改出一份更对得上的简历再投。
- **目标 B · 复盘（防守）**：对已投递岗位的结果（不合适 / 已读不回 / 面试 / offer）做回溯，反推问题、量化"改简历有没有用"。

**定性目标**
- G1（目标 A）：用户看到新 JD 时，能明确说出"我匹不匹配、差在哪、该怎么改简历"。
- G2（目标 B）：用户能明确说出"我的问题集中在 X 类岗位 / X 项能力"。
- G3（目标 B→A）：用户拿到"被拒 JD vs 我的简历"的具体差距清单与可替换修改段落，并据此产出下一版简历。

**定量目标**
- G4：同一批岗位类型下，新简历版本（A/B）回复率 ≥ 旧版本。

**非目标（本期不做）**
- 不改主仓"发现职位 / 智能匹配 / 生成简历"主流程。
- 不做自动代投。
- 不新增海外/国内采集平台（仅补 OCR 输入 + 爬虫占位接口）。

## 6. 详细功能需求

### FR1 多源录入"我的投递结果 / 新 JD"
- **描述**：支持 4 种录入适配器，统一产出标准字段字典后入库。**同一入口服务两条主线**——已投递结果走常规状态（rejected/no_response/...），新看到、尚在评估的 JD 走 `prospect`（待分析/观望）状态，先进来分析匹配度与差距，再决定是否改简历去投。
- **适配器**：
  - `add` 交互式逐条录入（含 `status` 字段，新 JD 填 `prospect`）；
  - `import-csv` 批量导入（CSV 模板见 `csv_template/`）；
  - `image` 截图 OCR（打印"贴给 WorkBuddy"提示词 → 视觉模型抽字段 → `--json` 回填入库）；
  - `scraper` 爬虫占位接口（不实现具体爬取，仅展示扩展位）。
- **规则**：`company` 与 `role` 为必填；入库同时写归档 `data/applications/<company>_<role>/`。`prospect` 记录不参与目标 B 的"已投递复盘"统计与 A/B 回复率计算，仅在目标 A 链路中被 `score --profile` 与 `revise` 引用。

### FR2 五维匹配度评分
- **描述**：对每条投递记录打五维分，加权出 `fit_overall`，写入记录。
- **五维**：技能 / 经验 / 文化 / 地点(PASS/FAIL/FLAG) / 职业契合，各 0–100。
- **权重**：技能 30% · 经验 25% · 文化 15% · 职业 30%；地点不参与加权。
- **模式**：`--mode heuristic`（离线默认，关键词重叠代理）/ `--mode llm`（直连，需 key）/ `--export-prompt`（打印贴 WorkBuddy 的提示词）。
- **规则**：`--all` 默认只评 `fit_overall` 为空者，避免覆盖 LLM/人工预填精准分。

### FR3 结果看板（统计）
- **描述**：按 `last_status` / `platform` / `role_type` / `sector` 分组计数，输出占比。

### FR4 fit×outcome 交叉分析
- **描述**：按 fit 分档（Strong 75+ / Good 60–74 / Moderate 45–59 / Weak 30–44 / Poor <30）× 状态交叉计数。
- **高亮规则**：
  - 强信号（rejected）+ Strong/Good → "高匹配却被拒：简历没讲清匹配点"；
  - 弱信号（no_response）+ Strong/Good → "高匹配却沉默：别急着大改"。

### FR5 JD 差距分析
- **描述**：对 rejected / no_response 记录，拉 JD + 当时简历版本，产出差距清单提示词。
- **模式**：默认打印"贴给 WorkBuddy"提示词（免 key）；`--mode llm` 预留直连。
- **规则**：诚实不虚构；已读不回为弱信号，不据此大改简历。

### FR6 简历版本 A/B 回复率
- **描述**：按 `resume_version` 分组，算投递数、正反馈数、回复率。
- **正反馈** = interview / offer / hired / offer_declined；**回复率** = 正反馈数 / 投递数。
- **对比**：取最早与最新版本，输出回复率差值与"有效/无效"判定。

### FR7 对照 JD 修改简历
- **描述**：针对被拒/已读不回且记录简历版本的岗位，基于 JD 差距生成修改草稿。
- **模式**：`export`（贴 WorkBuddy 提示词）/ `heuristic`（本地规则草稿，免 key）/ `llm`（Drafter 起草 + Reviewer 独立审查，需 key）。
- **产出**：`revised_resume.md` 写入归档目录，不覆盖原简历。

### FR8 简历 LaTeX 导出 + ATS 校验
- **描述**：从简历版本生成 `.tex`（极简模板、文本层干净）；可选 `--compile` 出 PDF；`--ats` 算关键词覆盖率。
- **ATS**：纯 Python 关键词覆盖率（JD vs 简历）+ 缺失清单；若有 PDF 且装 `pdftotext`，额外查文本层乱码（cid 标记）。
- **降级**：无 TeX 引擎只出 `.tex`；无 `pdftotext` 跳过文本层检查，不报错。

### FR9 原简历上传与解析（doc/pdf/txt → 文本）
- **描述**：提供"上传原简历"入口，接受 `.doc` / `.docx` / `.pdf` / `.txt` / `.md` / `.tex` 文件，解析为纯文本后存入 `resumes` 表作为**基准版本（原简历）**。它是 FR2（`--profile`）、FR5、FR7 的原材料前提——没有原简历就无法指出简历不足、也无法对照 JD 改简历。
- **解析策略（系统工具优先 + 双通道降级）**：
  - `.txt/.md/.tex`：直接读为文本；
  - `.pdf`：优先 `pdftotext`（poppler，可选依赖）；无则降级提示；
  - `.doc/.docx`：优先 macOS `textutil` 或 `libreoffice --headless --convert-to txt`（系统自带/可选）；无则降级；
  - 无任何可用解析器 → 走"导出提示词"双通道：打印"把简历文本贴给 WorkBuddy 提取"提示词，由用户回填文本（保持免 key 可跑）。
- **版本约定**：上传的原简历记为基准版本（默认 `orig`，可用 `--version` 指定）；revise 以其为"原简历"基准与 JD 比对，产出修改草稿，不覆盖原简历。
- **规则**：解析后写 `resumes(source_file, source_format, text, change_log)`，不改动已存的其他简历版本。

## 9. 约束与依赖

- **技术约束**：MVP 零重依赖（仅 stdlib `sqlite3`/`argparse`/`re`）。可选增强：`openai`（LLM 模式）、`pdftotext`（PDF 文本层）、`lualatex`（PDF 编译）。
- **合规约束**：脉脉不支持 CDP，仅 OCR/手动；爬虫只抓"自己的投递记录"，遵守平台条款。
- **隐私约束**：`data/` 个人数据经 `.gitignore` 排除，不入库、不上云。

## 10. 双通道策略（全局）

"双通道"是本产品的核心工程策略，贯穿 FR2/FR5/FR7：

| 通道 | 触发 | 是否需要 key | 用途 |
|---|---|---|---|
| **导出提示词（export）** | 默认 | 否 | 打印结构化提示词，贴给 WorkBuddy 完成分析/评分/修改 |
| **本地启发式（heuristic）** | 显式 | 否 | 离线规则产出基线结果（评分/修改草稿） |
| **LLM 直连（llm）** | 显式 `--mode llm` | 是（OPENAI_API_KEY） | 一键全自动，调用 OpenAI 兼容接口 |

设计意图：**没 key 也能跑通全文链路**，有 key 时升级为全自动；降低使用门槛，保护隐私。
